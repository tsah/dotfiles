# Local worktree workflow

## Architecture

Worktrunk is authoritative for worktree creation, setup hooks, switching, and destruction. Stock `wt` remains the native Worktrunk executable: this repository intentionally provides no `bin/wt`, compatibility wrapper, or parallel worktree manager. `alt-k-tui/src/workflow.ts` invokes native Worktrunk and adds only tmux and local agent orchestration; `bin/dotfiles-workflow` is its stable shell boundary.

A worker is identified by its canonical worktree path. Repository identity comes from `git rev-parse --git-common-dir`. Tmux session names are display labels (`repo@branch`) with an eight-character path hash only when that label collides. The canonical path and common dir are recorded as tmux user options. Session lookup scans those canonical path tags before generating a name, so renaming a session with tmux does not detach its agents or cause orchestration to create a duplicate canonical session. Legacy untagged sessions are adopted by canonical `session_path` and tagged on first use. Never use a tmux name as identity.

Sessions are created lazily. Their stable `main` window is retained; every visible agent gets a dedicated, uniquely named window that remains available for follow-up after the agent settles. `worker-pi`, `worker-claude`, and `worker-opencode` are thin harness adapters. `agent-*` starts another visible agent in the current worktree. OpenCode is not filesystem-sandboxed. Repository agent configuration takes precedence over personal configuration through each harness's native lookup.

Normal Pi delegation uses the pinned `@tintinweb/pi-subagents` package. Its `Agent`, `get_subagent_result`, and `steer_subagent` tools run child sessions in the parent Pi process, subject to `pi/subagents.json`; they do not create tmux windows and do not appear in Alt-K. Its native `isolation: worktree` mode bypasses Worktrunk, so use `worker-pi` instead whenever a task needs an isolated handoff or visible session.

The Alt-K picker is the shared interactive entrypoint (`workspace-picker`, and shell function `s`). It combines tmux sessions, worktrees/configured directories, git state, and agent state. Its jump screen is a bottom-sorted selectable tree: tmux sessions are compact parents, trees with up to three window/agent children begin expanded, and larger trees begin collapsed. Session, picker, active-pane, and agent events persist canonical-path activity under `XDG_STATE_HOME`; after cache, tmux-server, or machine restart, sessionless directories sort by the newest durable event, dirty-file mtime, worktree reflog, or commit, with zoxide frecency as a final tie-breaker. Directory rows display the winning source and age, and expensive Git recovery evidence refreshes at most once per minute. Left and Right collapse or expand a session; filtering a child retains its parent context even when collapsed. When opened from tmux, the current session parent is initially selected. Enter jumps to the selected session, window, agent, worktree, or directory. Agent state is normalized as `blocked`, `working`, `done`, `idle`, or `unknown` and displayed as color-coded `waiting`, `working`, `ready`, `idle`, or `unknown`; a completion remains ready until that target is focused, then becomes idle, while a newer completion becomes ready again. Alt-D asks for y/n confirmation and destroys an exact selected child pane; if that pane is the session's last pane, it destroys the session and its linked worktree instead. Alt-R renames only a selected tmux session parent by stable session ID while leaving canonical path metadata and included agents unchanged. Pressing Alt-K again while the picker is open begins the explicit repository picker, then presents existing worktrees, local branches, and remote branches together. Selecting a repository renders cached refs immediately while `git fetch --all --prune` refreshes them in the background; remote rows are sorted by commit recency. Text input and bracketed paste both work in search and branch/base fields. Typing a name without an exact branch match offers to create that branch and prompts for its base. The flow delegates worktree creation and switching to native Worktrunk, then lazily ensures the tmux session. Alt-B opens the same UI at the repository step. Worktree creation/destruction remains delegated to native Worktrunk so its setup and confirmation behavior remains authoritative.

The Pi lifecycle extension is installed globally so Alt-K receives status from every Pi process running in tmux, including Pi sessions started directly rather than through `worker-pi`. Pi settled waiting uses `agent_settled`, not `agent_end`, and writes the result signal without closing the agent window. A timeout reports failure and leaves the window and worktree running. Neovim considers only agent windows tagged with the canonical current worktree path. It auto-targets exactly one; otherwise it uses `vim.ui.select`. Sending uses tmux's acknowledged buffer APIs. It never reads pane screen contents or retries blind.

## Milestones and acceptance criteria

1. **Identity and orchestration:** canonical path/common-dir identity; lazy collision-safe session; stable main window; multiple agent windows; thin harness adapters; native `wt switch --create` setup.
2. **One navigation surface:** Alt-K and `s` invoke one picker; picker displays sessions, configured/zoxide directories, git and agent status; no legacy switcher or remote-worker process remains.
3. **Editor and agent lifecycle:** unified `Agent*` commands, current-worktree filtering, explicit unsaved-buffer choice, settled Pi wait, verified self-destruction boundary.
4. **Distribution:** one desktop/server manifest, pinned Pi subagent package installation, only ledger-owned stale links reconciled, neutral skills under `skills/`, obsolete entrypoints absent.
